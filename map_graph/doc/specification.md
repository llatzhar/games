Specification
=============

# 1. 目的とスコア

- 敗北せずに生存したターン数をスコアとする。
- 敵国の都市を占領することを目的としてプレイするが、敵都市は追加され続けるため勝利条件ではない。

# 2. 対象と技術

- プラットフォーム: Windows ローカル実行のみ。
- 言語: Python。
- 描画/入出力: pyxel。
- オフラインに閉じたシステム。

# 3. 画面と操作

- 操作: キーボード + マウス。
- 画面: メインのマップ表示 + ゲーム内ログ表示スペース。

# 4. ターン進行

- 自軍(プレイヤー)が n 回行動すると、敵軍(CPU)が m 回行動する。
- n と m はゲーム中は固定。
- 難易度によって n/m が変わる可能性がある。

## 4.1 コアゲームループ

1. プレイヤーターン
   1. コマンド実行(ユニット移動 | ユニット雇用)
   2. 戦闘処理(兵力 0 の軍ユニットは消失)
   3. コマンド数が残っていれば 1-1 へ
2. マップ処理(占領実行、新都市発見)
3. CPUターン
   1. コマンド実行
   2. 戦闘処理
   3. コマンド数が残っていれば 3-1 へ
4. マップ処理(CPUユニットによる都市占領)

# 5. マップ

- 都市と都市が道路でつながるグラフ。
- 初期状態: 自軍 2 都市、敵軍 1 都市を正三角形に配置。
- 新しい都市は既知の都市と道の外側に生成される。
- 新しい都市から既知の都市に 1-3 本の道を生成する。
- 新しい道は既存の道と交差しない。
- 道の長さは一定以内に制限する。

# 6. 都市、人口、徴兵

- 都市は生産力と都市規模(人口上限)を持つ。
- 人口は毎ターン、都市規模(人口上限)の 10% ずつ増加し、上限まで増える。
- ユニット提供のタイミング:
  1. 初期都市には最初からユニットが付与される。
  2. 都市の占領完了時にユニットを提供する。
  3. 都市に所属ユニットがいないときに徴兵コマンドを行う。
- 徴兵コマンドで生成されるユニット兵数は、徴兵時点の都市人口と同じ値とする。
- 徴兵実行で人口は一時的に 0 になる。

# 7. ユニットとリーダー

- 軍ユニットは兵数を持ち、兵数が 0 になると消失する。
- リーダーは個別のステータスを持つ。
- 軍ユニットはリーダーに率いられ、性能にリーダー補正が乗る。
- 自軍が他国の都市を解放するとリーダーを 1 人獲得する。
- 同陣営のユニットは 1 都市に複数滞在でき、上限は設けない。

# 8. 移動

- 1 コマンドで隣接都市へ 1 回移動。

# 9. 戦闘

- 都市に自軍と敵軍が同時滞在すると戦闘が発生する。
- 戦闘は同時損耗。
- 1 ラウンドのダメージ計算:
  - 基本値: 兵数 / 10 (定数)
  - リーダー攻撃補正を乗算
  - 乱数幅: -10% から +10%
  - 結果を相手兵数から減算
- 5 ラウンド終了 or どちらかの兵数が 0 になるまで継続。
- 複数ユニットが同一都市で戦闘する場合:
  - 両陣営のユニット兵数を合算して計算する。
  - ユニット数が多い方が有利になるよう、1 ユニットあたり攻撃力 +10% を付与する。

# 10. 占領

- 都市に他国の軍が滞在すると占領が始まる。
- 都市の占領耐久は都市の現人口を初期値とし、0 以下で占領完了。
- 占領耐久は占領値を毎ターン減ずる。
- 占領値は都市に滞在するユニット数の合計に、リーダー占領補正を乗じる。
- リーダーが占領補正を持つ場合、兵数に占領補正を乗じる。
- 1 ターンで完了しない場合は持ち越す。
- 完了すると都市の所属が変更される。

# 11. CPU強化と難易度

- CPUはプレイヤー都市の占領とプレイヤーユニット撃破を目標に行動する。
- CPUは行動回数分だけユニット移動を行う。
- CPUは既定ターン毎に新しく都市が生成される。
- 生成時の所属ユニット数がターン経過で増えていく。
- ゲーム内でターンが進むにつれて上がる難易度調整:
  - 新都市頻度
  - 敵ユニット数
  - 敵リーダー補正
- ゲーム開始時に選ぶ難易度調整:
  - CPU行動回数
  - 人口成長
- 強化式はパラメータ化して調整可能にする。

# 12. ログ

- ログ対象: 戦闘 / 生成 / 移動 / 占領。
- 画面上にログ表示スペースを持つ。

# 13. 永続化

- 保存対象: マップ(敵味方ユニット状態含む)、ターン、ログ履歴。
- 自動保存のみ。
- モデル/ゲームオブジェクトは JSON でシリアライズする。

# 14. 設計方針

- 画面遷移は階層化 StateMachine を採用。
- モーダルダイアログのために Pushdown の仕組みを持つ。

# 15. 未確定/要調整項目

- CPU行動ロジックの具体化。
- 難易度/強化の数式と初期パラメータ。
- リーダー補正値の具体レンジ。
