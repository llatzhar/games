# ADR-002: 道路生成における交差回避戦略

## Status

**Proposed** - 2026-01-28

---

## Context

新都市を追加する際、既存都市との道路を1〜3本生成する。README.mdの要件として「新しい道は既存の道と交差してはならない」が明記されている。

### 制約条件

1. 新都市から既存都市へ1〜3本の道路を生成
2. 新道路は既存道路と交差してはならない
3. 道路長には上限がある（マップ把握性のため）
4. 最低1本の道路は必ず生成される必要がある（孤立都市禁止）

### トレードオフ

- 交差禁止を厳格に守ると、道路が引けない状況が発生しうる
- 道路本数が少ないと防衛しやすいが攻略ルートが限定される
- 道路本数が多いとゲーム性は上がるが交差リスクも上がる

---

## Decision

**「最近接優先 + 交差判定 + フォールバック強制接続」方式を採用する**

### アルゴリズム概要

1. 新都市から全既存都市への距離を計算し、近い順にソート
2. 近い都市から順に道路候補を生成
3. 各候補について:
   - 道路長が上限以内か確認
   - 既存全道路との交差判定
   - 今回生成済みの新道路との交差判定
4. 交差なしなら道路を採用
5. 目標本数に達するか、候補を使い切るまで繰り返し
6. **道路が0本の場合、最近接都市への道路を強制生成**（交差許容）

---

## Considered Options

### Option A: 完全ランダム接続

```
接続先 = ランダムに1〜3都市を選択
交差チェック = なし
```

**長所:**
- 実装が最もシンプル
- 予測不能なマップ形状

**短所:**
- 交差が発生し、視覚的に混乱するマップになる
- 道路が絡み合うと都市選択時の誤操作が増える

**却下理由:** README.mdの「交差禁止」要件を満たさない。

---

### Option B: 厳格な交差禁止（フォールバックなし）

```
道路生成ループ:
  if 交差なし and 距離制限内:
    道路追加
  else:
    スキップ
結果: 0〜3本の道路（0本もありうる）
```

**長所:**
- 交差が絶対に発生しない
- アルゴリズムがシンプル

**短所:**
- **孤立都市が発生する可能性がある**
- 孤立都市はゲーム進行を破綻させる（到達不能な敵）

**却下理由:** ゲームプレイの根本を破壊するリスクがある。

---

### Option C: Delaunay三角形分割

```
全都市でDelaunay三角形分割を実行
三角形の辺を道路として採用
```

**長所:**
- 数学的に交差しないことが保証
- 均一な道路ネットワークが生成される

**短所:**
- 毎回全体を再計算する必要がある
- 既存道路を維持しながらの差分更新が困難
- 都市追加で既存道路が消える可能性（プレイヤー混乱）
- 実装コストが高い

**却下理由:** インクリメンタル生成との相性が悪い。

---

### Option D（採用）: 最近接優先 + 交差判定 + フォールバック

```
candidates = sort_by_distance(existing_cities, new_city)
roads = []

for candidate in candidates:
    road = create_road(new_city, candidate)
    if road.length <= MAX_LENGTH and not intersects_any(road):
        roads.append(road)
        if len(roads) >= target_count:
            break

if len(roads) == 0:
    # フォールバック: 最近接に強制接続
    roads.append(create_road(new_city, candidates[0]))
    log_warning("Forced road with potential intersection")

return roads
```

**長所:**
- 通常ケースでは交差しない
- 孤立都市が発生しない（ゲーム進行保証）
- フォールバック発生は稀で、発生時もログで検知可能
- インクリメンタル生成と相性が良い

**短所:**
- 理論上は交差が発生しうる（フォールバック時）
- フォールバック発生を最小化するパラメータ調整が必要

---

## Consequences

### 体験面

| 影響 | 詳細 |
|-----|------|
| ✅ 視認性 | 通常は交差なしでクリーンなマップ |
| ✅ プレイ継続性 | 孤立都市が発生せず、すべての敵に到達可能 |
| ⚠️ レアケース | フォールバック交差時、視覚的な違和感が発生しうる |
| ✅ 戦略性 | 近い都市から優先接続されるため、地理的に意味のあるネットワーク |

### 技術面

| 影響 | 詳細 |
|-----|------|
| ✅ 計算量 | O(C × R) - 都市数C、道路数R。許容範囲内 |
| ✅ 実装容易性 | 線分交差判定は標準的なアルゴリズム |
| ⚠️ デバッグ | フォールバック発生を検知・記録するログ機構が必要 |
| ✅ 調整容易性 | `max_roads`, `max_length` で挙動調整可能 |

### 将来の拡張

- 交差発生時の道路視覚表現（橋・トンネル表現）を追加可能
- 「戦略的重要都市」への優先接続ルール追加可能
- 地形に沿った道路生成への拡張可能

---

## 補足: 線分交差判定アルゴリズム

採用するのはCCW（Counter-Clockwise）判定に基づく方法：

```python
def segments_intersect(seg1, seg2):
    """
    seg1: (A, B), seg2: (C, D)
    端点共有は交差とみなさない
    """
    def ccw(P, Q, R):
        return (Q.x - P.x) * (R.y - P.y) - (Q.y - P.y) * (R.x - P.x)
    
    A, B = seg1
    C, D = seg2
    
    # 端点共有チェック
    if A == C or A == D or B == C or B == D:
        return False
    
    d1, d2 = ccw(A, B, C), ccw(A, B, D)
    d3, d4 = ccw(C, D, A), ccw(C, D, B)
    
    return (d1 * d2 < 0) and (d3 * d4 < 0)
```

この方法は:
- 計算が軽量（乗算・減算のみ）
- 数値誤差に強い
- 端点共有を正しく除外できる
